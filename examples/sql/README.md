# sql — Example

A runnable example showing how `@playwright-labs/fixture-sql` and `@playwright-labs/ts-plugin-sql` work together in Playwright tests.

## What this example covers

| Feature | Where to look |
|---------|--------------|
| Auto-managed SQLite connection per test | `src/tests/example.spec.ts` |
| `sql` tagged template — INSERT / SELECT / JOIN / UPDATE / DELETE | `src/tests/example.spec.ts` |
| Parameterised queries with `?` placeholders | `src/tests/example.spec.ts` |
| Aggregate functions (`COUNT`) | `src/tests/example.spec.ts` |
| TypeScript plugin — **inline schema** (column autocomplete in editor) | `tsconfig.json` → first plugin entry |
| TypeScript plugin — **schemaFile** (types from `db-types.ts`) | `src/tests/example-schema-file.spec.ts` |
| Row type interfaces generated by `fixture-sql pull` | `src/db-types.ts` |

## Quick start

```sh
# From the monorepo root:
pnpm --filter sql-example test

# Or from this directory:
pnpm test
```

No database server needed — every test runs against an in-memory SQLite database that is created and torn down automatically.

## How the TypeScript plugin works

The `@playwright-labs/ts-plugin-sql` language service plugin runs inside VS Code / your editor and provides:

- **Column autocomplete** — type `SELECT ` inside `sql\`...\`` and see all column names for the referenced table
- **Table autocomplete** — type `FROM ` or `JOIN ` to see available tables
- **Structural diagnostics** — missing `FROM`, `SET`, or other required clauses are underlined in red
- **Hover types** — hover a column name to see its TypeScript type

Two configuration approaches are shown side by side:

### Approach 1 — inline schema (`tsconfig.json`)

```json
{
  "name": "@playwright-labs/ts-plugin-sql",
  "tag": "sql",
  "schema": {
    "users": { "id": "number", "name": "string", "email": "string", "created_at": "string" },
    "posts": { "id": "number", "user_id": "number", "title": "string", "published": "boolean" }
  }
}
```

Quick setup for small projects — schema lives directly in `tsconfig.json`, no extra files needed.

### Approach 2 — `schemaFile` (`src/db-types.ts`)

```json
{
  "name": "@playwright-labs/ts-plugin-sql",
  "tag": "sql",
  "schemaFile": "./src/db-types.ts"
}
```

The plugin reads `src/db-types.ts` via the TypeScript AST and derives completions from the exported `Tables` type. In a real project, generate this file with:

```sh
pnpm --filter @playwright-labs/fixture-sql pull -a sqlite -u ./example.db -o src/db-types.ts
```

## Fixtures used

### `sql` fixture

Provides a connected database adapter for the current test. The connection is opened before the test and closed automatically after — no manual setup or teardown needed.

```typescript
test("query users", async ({ sql: db }) => {
  const { rows } = await db.query<{ name: string }>(
    sql`SELECT name FROM users WHERE email = ?`,
    ["alice@example.com"],
  );
  expect(rows[0]!.name).toBe("Alice");
});
```

### `test.use({ sqlAdapter })`

Selects which database backend to use. This example uses SQLite in-memory:

```typescript
import { sqliteAdapter } from "@playwright-labs/fixture-sql/sqlite";

test.use({ sqlAdapter: sqliteAdapter(":memory:") });
```

Switch to a real file by passing a file path, or use the PostgreSQL / MySQL adapters for those backends.
